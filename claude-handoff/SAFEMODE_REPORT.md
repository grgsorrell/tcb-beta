### The Safe Mode system, end to end

**What sets the flag.** `getValidatorFiringBreakdown(conversation_id)` (worker.js ~683) counts rows
with `action_taken = 'stripped'` across the six per-fact-class validator tables (`sam_validation_events`
[geographic], `sam_compliance_validation_events`, `sam_finance_validation_events`,
`sam_donation_validation_events`, `sam_opponent_validation_events`, `sam_citation_validation_events`)
for the conversation. `safeModeActive = total >= SAFE_MODE_THRESHOLD` (**= 5**; the code comment still
says "3+", stale). Only `'stripped'` counts â€” regenerated/tagged/passed are excluded by design. So Safe
Mode = "this conversation had â‰Ą5 validator STRIPS."

**Where it's stored.** Two places, both keyed by `conversation_id` (i.e., per session):
1. The **count is recomputed live every turn** from the `sam_*_validation_events` tables (which are
   append-only and never pruned within a conversation).
2. `sam_safe_mode_events` holds one activation row per conversation (`INSERT OR IGNORE`), with the
   activation turn number stashed inside the `triggering_validator_breakdown` JSON (used only for
   banner cadence).

**What clears it â€” NOTHING (hypothesis confirmed).** Grep for `DELETE/UPDATE`/decay/reset against
`sam_safe_mode_events` and the validator tables returns nothing. Because the count is derived from
append-only strip events filtered by `conversation_id`, once a conversation hits 5 strips **every
subsequent turn in that conversation is in Safe Mode permanently.** The only "clear" is starting a new
session (new `conversation_id`). There is no clean-turn decay and no time decay.

**Where it alters output (two places):**
1. **Stricter prompt block** (~7291): when `safeModeActive`, a "SAFE MODE ACTIVE â€” RELIABILITY

   of that block â€” to *say* "I want to be careful here â€” I've had some accuracy issues in our
   conversationâ€¦". So Safe Mode literally instructs Sam to **name-drop its own degraded state** to the
   candidate.
2. **Mechanical banner** (`applySafeModeBanner`, ~6084, called from `buildSafeResponse` ~6172): prepends
   one of six rotating disclaimer variants â€” including the reported "Note: campaign rules change between
   cyclesâ€¦" (SAFE_MODE_BANNERS[5], ~6045) â€” to the first text block of the delivered response.

### Explaining the three observed symptoms

**(a) Disclaimer on EVERY turn, including conversational ones.** Two compounding causes:
- **Category-blind.** `applySafeModeBanner` never checks `_questionCategory`. It fires on any turn
  where `safeModeActive` + the cadence gate passes â€” greetings, "thanks", "what's next?" included.
- **Brittle cadence.** The intended cadence is "activation turn, then every 4 turns," computed as
  `_safeModeTurnNumber - _safeModeActivationTurn`, where `_safeModeTurnNumber = history.length`.
  `history.length` is a noisy proxy: it grows by â‰Ą2 per user turn (user + assistant) and by more when a
  turn involves tool-call rounds (tool_use + tool_result entries). With that noisy step size, `delta %
  4 === 0` degrades from "every 4 turns" to firing on most/every user turn. Combined with "nothing
  clears Safe Mode," the disclaimer became effectively persistent.

**(b) Disclaimer TWICE in one response.** There are **two independent disclaimer sources**, and both can
appear in the same delivered turn:
- the **mechanical banner** prepended by `applySafeModeBanner`, and
- Sam's **own model-generated caution**, which the Safe Mode stricter-prompt block explicitly tells him
  to write ("I want to be careful here â€” I've had some accuracy issuesâ€¦").
  The banner and the model's self-caution are generated by different layers that don't know about each
  other, so the candidate sees two "verify with your authority / rules change between cycles"
  disclaimers stacked. (The `isMultiRoundFollowUp` guard only prevents the *mechanical* banner from
  double-firing across tool rounds; it does nothing about the prompt-induced second disclaimer.)

**(c) Why it activated in this session â€” the Georgia regen.** Safe Mode needs 5 `'stripped'` events. The
Georgia turn's tell â€” an answer opening with an unexplained "My apologies for the oversight," â€” is the
signature of a **validator stripâ†’regenerate**: the regen prompt starts with "STOP. Your previous
response contained unverified content the validator caught," and the regenerated answer apologizes and
comes back vaguer. So a citation/compliance validator STRIPPED content on the Georgia turn (and enough
similar strips across the session reached the threshold of 5), flipping the conversation into Safe Mode.
The deep root cause of *why* a correct `ethics.ga.gov` citation was stripped is **item 4 (Group B)** â€”
investigated and fixed there.
